# RAG 정밀도 개선 면접 답변 정리

## 1) "정확도를 어떻게 높였나요?"

짧은 답변:

> "초기에는 키워드/유사도 기반이라 문서/설정 파일이 섞였습니다. 그래서 인덱싱 제외 규칙, 서비스/경로 필터, 한국어 동의어 확장, BM25 재정렬을 단계적으로 넣어 정확도를 개선했습니다."

핵심 4단계:

1. **인덱스 클린업**: vendor/node_modules/개인문서 제외  
2. **스코어 튜닝**: `.go` 우대, 문서/설정 페널티  
3. **재정렬**: 후보군에 BM25 적용  
4. **하드 필터**: `--service`, `--path-prefix`로 검색 범위 강제

---

## 2) "BM25는 왜 넣었나요?"

> "벡터 검색은 의미 유사도는 강하지만, 구현 위치 찾기 질문에서는 키워드 정확도가 중요합니다. BM25를 재정렬 단계에 추가해 질문 핵심 단어가 실제 코드에 있는지를 더 강하게 반영했습니다."

---

## 3) "오답을 어떻게 줄였나요?"

> "오답의 주 원인은 문맥 오염이었습니다. 그래서 검색 범위를 강제하는 옵션을 제공했습니다. 예를 들어 ORDERFC 질문은 `--service ORDERFC`로 제한해 다른 서비스 코드가 상위로 올라오지 않게 했습니다."

---

## 4) "단순 챗봇과 무엇이 다르나요?"

> "질문을 그냥 생성 모델에 던지는 구조가 아니라, 코드 인덱싱 → 검색/재정렬 → 근거 기반 답변으로 나눴습니다. 답변에는 파일 경로를 포함해 검증 가능성을 유지했습니다."

---

## 5) "한계는 뭐였고 어떻게 대응했나요?"

한계:
- 한국어 질문의 표현 다양성(예: 멱등성/중복방지)
- 넓은 토큰(`token`)으로 인한 노이즈

대응:
- 동의어 확장 규칙 추가
- 구현 질문 시 코드 파일 하드 필터
- 서비스/경로 옵션으로 운영 시 정확도 우선

---

## 6) 30초 답변 스크립트

> "코드 Q&A RAG를 만들면서 정확도를 단계적으로 개선했습니다. 초기엔 문서/설정이 섞여 오답이 나와서, 인덱스 제외 규칙과 `.go` 우선 가중치를 넣고, 1차 검색 뒤 BM25 재정렬을 추가했습니다. 또 실무처럼 `--service`, `--path-prefix` 필터로 검색 범위를 강제해 문맥 오염을 줄였습니다. 결과적으로 답변 품질뿐 아니라 근거 경로 기반 검증 가능성을 확보했습니다."
